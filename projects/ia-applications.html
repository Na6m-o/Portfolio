01 / CONTEXTE

Pourquoi l'IA ?

L'intelligence artificielle est en train de transformer le metier de l'IT. Les outils de generation de code, les assistants, les LLMs — ce n'est pas une menace, c'est un levier. Ceux qui savent s'en servir sont deux fois plus efficaces.

J'ai decide de ne pas subir cette evolution mais d'en prendre le controle. En construisant mes propres outils bases sur l'IA, je comprends les mecanismes, je decouvre les limites, et je developpe des applications qui m'aident concretement dans mon quotidien IT.

Type Prototypage & experimentation
Statut En cours — veille active

02 / REALISATIONS

Ce que j'ai construit

BOT

Assistant IT IA

Prototype d'assistant conversationnel pour le support IT — pose une question sur une erreur systeme, un code d'evenement Windows, ou un probleme reseau, et obtient une reponse structuree avec des pistes de resolution.

  • Interface simple (HTML/JS)
  • API LLM en backend
  • Contexte IT pre-configure
  • Suggestions de commandes
GEN

Generateur de scripts

Outil qui, a partir d'une description en langage naturel ("cree un script qui desactive les comptes inactifs depuis 60 jours"), genere le script PowerShell ou Bash correspondant avec commentaires et gestion des erreurs.

  • Description naturelle → code
  • PowerShell & Bash
  • Commentaires auto-generes
  • Validation basique du code
DOC

Analyseur de logs

Outil experimental qui prend en entree un extrait de logs systeme ou reseau et produit une analyse lisible : resume des evenements, detection d'anomalies potentielles, suggestions d'actions.

  • Paste de logs → analyse
  • Detection patterns anormaux
  • Resume en francais clair
  • Recommandations d'actions
03 / METHODOLOGIE

Comment je construis

01

Identifier un probleme reel

Chaque outil que je construis part d'un vrai point de friction rencontre dans mon travail quotidien. Je ne construis pas pour construire — je construis parce que quelque chose me fait perdre du temps ou m'est difficile a expliquer/trouver.

02

Prototyper vite

Un prototype fonctionnel vaut mieux qu'une architecture parfaite sur le papier. Je commence par la version la plus simple possible — souvent une interface HTML/JS basique qui appelle une API — et j'itere.

03

Tester en conditions reelles

Je teste les outils sur de vrais problemes, avec de vrais logs, de vrais scenarios. C'est la que les limites apparaissent et que l'outil s'ameliore vraiment.

04

Documenter et partager

Chaque prototype fait l'objet de notes : ce qui fonctionne, ce qui ne fonctionne pas, les idees d'amelioration. Documenter pour soi, c'est documenter pour apprendre.
